# Codex Prompt Pack for Mobile Developer Portfolio

Use these prompts inside the Codex IDE extension or Codex CLI while this repo is open in VS Code.

## Working rules for Codex

Paste this first when you start a session:

```text
You are working inside a GitHub Pages portfolio repository for a mobile developer.
Keep the site static HTML, CSS, and vanilla JavaScript only.
Do not add build tools, package managers, frameworks, or external dependencies unless explicitly requested.
Maintain GitHub Pages compatibility.
Preserve accessibility, responsiveness, and clean semantic HTML.
Prefer small, reviewable commits.
When making visual changes, update copy and layout together so the site feels intentional.
```

---

## 1) Upgrade the visual design

```text
Audit this portfolio site and upgrade it to feel more premium, modern, and recruiter-friendly.
Keep it static HTML/CSS/JS with no build step.
Improve typography, spacing, hierarchy, button styling, section separation, and project card polish.
Preserve responsiveness and GitHub Pages compatibility.
Show me the exact files changed and briefly explain the design rationale.
```

## 2) Tailor the hero for mobile development roles

```text
Refactor the hero section so it is clearly targeted at mobile software engineering roles.
Emphasize Android, iOS, UI implementation, debugging, API integration, and applied technical problem solving.
Keep the tone professional and concise.
Add a stronger primary CTA and a secondary CTA.
```

## 3) Replace placeholder content with recruiter-facing copy

```text
Rewrite the site copy to sound like a recent Computer Science graduate specializing in Mobile Development with prior RF/electronics troubleshooting experience.
Keep claims credible and specific.
Avoid buzzword stuffing.
Focus on Android first, with supporting iOS experience.
Update the About, Skills, Experience, and Project sections.
```

## 4) Improve project cards

```text
Upgrade the project section into stronger portfolio case-study cards.
For each project, include: purpose, stack, what I built, and one short impact/result line.
Keep the layout compact and easy to scan on mobile and desktop.
Do not introduce frameworks.
```

## 5) Add a resume call-to-action

```text
Add a dedicated resume call-to-action section near the top of the page and another near the footer.
Use placeholder links if needed.
Style both so they look intentional and not repetitive.
```

## 6) Add screenshots or mockup placeholders

```text
Update the project section to support screenshots or device mockups for each project.
Use semantic HTML and lightweight CSS.
If image assets are missing, create tasteful placeholder containers and comments showing where image files should go.
```

## 7) Improve navigation and scrolling behavior

```text
Polish the navigation for a single-page portfolio.
Add active-section highlighting if appropriate, smooth scrolling, and a better mobile menu if needed.
Keep JavaScript simple and maintainable.
```

## 8) Add contact section improvements

```text
Refactor the contact section so it feels more professional and actionable.
Include GitHub, LinkedIn, email, and a resume link.
Make the section visually stronger without becoming flashy.
```

## 9) Accessibility and QA pass

```text
Perform an accessibility and QA pass on this portfolio site.
Check heading structure, button/link labels, color contrast risks, keyboard access, focus states, and responsive layout issues.
Fix any problems you find.
Summarize what you changed.
```

## 10) SEO and metadata pass

```text
Improve the SEO and sharing metadata for this static portfolio site.
Review the title tag, meta description, Open Graph tags, favicon usage, and social preview references.
Keep everything GitHub Pages compatible.
```

## 11) Make it feel more like a product engineer portfolio

```text
Refactor the portfolio so it feels less like a school project gallery and more like a product-minded mobile engineer portfolio.
Increase clarity around problem solving, implementation details, and platform skills.
Keep the design clean and professional.
```

## 12) Stronger experience section

```text
Rewrite the experience section so it better connects previous electronics/RF troubleshooting work to software engineering strengths.
Highlight debugging, root-cause analysis, technical documentation, quality, and cross-functional execution.
Keep it honest and concise.
```

## 13) Add a featured project section

```text
Create a featured project section above the rest of the projects.
Choose the strongest mobile project and give it a larger layout with stack, responsibilities, and notable implementation details.
Keep the overall page balanced.
```

## 14) Build a version for Android-heavy applications

```text
Create an alternate version of the portfolio copy that is optimized for Android developer roles.
Emphasize Kotlin, Java, Jetpack Compose, Android SDK patterns, background work, persistence, media workflows, and API usage where relevant.
Keep the structure the same so I can compare versions easily.
```

## 15) Build a version for iOS-heavy applications

```text
Create an alternate version of the portfolio copy that is optimized for iOS developer roles.
Emphasize Swift, UIKit, Storyboards, state handling, persistence, and mobile UI implementation.
Keep all claims accurate to the existing project background.
```
